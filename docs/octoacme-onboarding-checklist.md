# OctoAcme — Role Onboarding Checklist

## Purpose
This checklist ensures new team members are effectively onboarded to their role within OctoAcme projects. It provides a structured approach to familiarize new members with responsibilities, tools, processes, and team interactions.

---

## General Onboarding (All Roles)

### Week 1: Project & Team Orientation
- [ ] Complete OctoAcme project management documentation review
  - [ ] Read [Project Management Overview](octoacme-project-management-overview.md)
  - [ ] Review [Roles & Personas](octoacme-roles-and-personas.md)
  - [ ] Understand [RACI Matrix](octoacme-raci-matrix.md)
- [ ] Meet with Project Manager for project context and current status
- [ ] Introduction meeting with all team members
- [ ] Access granted to:
  - [ ] Project repository and documentation
  - [ ] Communication channels (Slack, Teams, etc.)
  - [ ] Project management tools (GitHub Projects, Jira, etc.)
  - [ ] Shared drives and documentation repositories
- [ ] Review current sprint/iteration goals and backlog
- [ ] Understand Definition of Done (DoD) and acceptance criteria standards

### Week 2: Integration & Contribution
- [ ] Shadow key team meetings (standups, planning, reviews)
- [ ] Review recent retrospective notes and action items
- [ ] Identify and schedule 1:1s with key collaborators
- [ ] Begin contributing to assigned tasks
- [ ] Understand escalation paths and communication cadence

### Ongoing: First 30 Days
- [ ] Participate in at least one retrospective
- [ ] Complete first significant deliverable in your role
- [ ] Provide feedback on onboarding experience
- [ ] Schedule 30-day check-in with Project Manager

## Role-Specific Onboarding

### Developer
- [ ] Set up local development environment
- [ ] Complete first code review (as reviewer)
- [ ] Submit first pull request
- [ ] Review coding standards and style guides
- [ ] Understand branching strategy and CI/CD workflow
- [ ] Meet with Technical Lead for architecture overview
- [ ] Access to deployment environments and logging tools

### Product Manager
- [ ] Review product roadmap and strategic priorities
- [ ] Access to analytics and user feedback tools
- [ ] Meet with key stakeholders (customers, leadership, partners)
- [ ] Review customer personas and market research
- [ ] Understand current success metrics and KPIs
- [ ] Review backlog prioritization framework
- [ ] Shadow or lead first stakeholder update meeting

### Project Manager
- [ ] Review active projects and their current status
- [ ] Access to all project tracking and reporting tools
- [ ] Meet with each role to understand their workflows
- [ ] Review existing risk registers and mitigation plans
- [ ] Understand budget and resource constraints
- [ ] Review communication templates and reporting cadence
- [ ] Facilitate or co-facilitate first team meeting

### QA Lead
- [ ] Set up testing environments and access to test data
- [ ] Review existing test strategies and test cases
- [ ] Access to bug tracking and test management tools
- [ ] Understand automated testing frameworks and CI/CD integration
- [ ] Review quality metrics and coverage reports
- [ ] Meet with Developers to understand testability practices
- [ ] Participate in or lead first defect triage meeting

### UX/UI Designer
- [ ] Access to design tools (Figma, Sketch, Adobe XD, etc.)
- [ ] Review design system and component library
- [ ] Access to user research repository and insights
- [ ] Understand design handoff process with Developers
- [ ] Review accessibility standards and guidelines
- [ ] Meet with Product Manager to understand user personas
- [ ] Shadow or conduct first usability testing session

### DevOps Engineer
- [ ] Access to cloud infrastructure and deployment platforms
- [ ] Review infrastructure as code repositories
- [ ] Understand current CI/CD pipelines and automation
- [ ] Access to monitoring, logging, and alerting systems
- [ ] Review incident response procedures and on-call rotation
- [ ] Meet with Security Champion to understand security requirements
- [ ] Shadow or participate in first deployment

### Security Champion
- [ ] Review security policies and compliance requirements
- [ ] Access to security scanning and vulnerability management tools
- [ ] Understand secure development lifecycle practices
- [ ] Review threat models and security architecture
- [ ] Meet with DevOps Engineer regarding infrastructure security
- [ ] Participate in security code review
- [ ] Review incident response and escalation procedures

## Onboarding Completion Checklist

### For New Team Member
- [ ] All general onboarding tasks completed
- [ ] All role-specific tasks completed
- [ ] Comfortable with tools and processes
- [ ] Clear understanding of responsibilities and accountabilities
- [ ] Established working relationships with key collaborators

### For Project Manager
- [ ] New team member has completed onboarding checklist
- [ ] 30-day check-in completed
- [ ] New team member is contributing effectively
- [ ] Feedback collected for onboarding process improvement
- [ ] RACI matrix updated if needed based on team changes

---

## Tips for Successful Onboarding

### For New Team Members
- Don't hesitate to ask questions—psychological safety is a core principle
- Take notes during meetings and document your learnings
- Schedule regular check-ins with your manager and key collaborators
- Seek opportunities to pair with experienced team members
- Share your fresh perspective—you may identify improvement opportunities

### For Team Members Supporting Onboarding
- Be proactive in offering help and context
- Share relevant documentation and resources
- Provide constructive feedback early and often
- Make time for pairing and shadowing opportunities
- Remember what it was like to be new—be patient and welcoming

---

## Continuous Improvement
This onboarding checklist should be reviewed and updated regularly based on:
- Feedback from newly onboarded team members
- Changes in tools, processes, or team structure
- Retrospective insights and lessons learned
- Evolving project needs and organizational practices
